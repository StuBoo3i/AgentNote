## 6. 代表性错误分析

### 6.1 表头不匹配但值正确

这类问题数量最多，说明 agent 计算正确，但提交 CSV 的列名与 evaluator 期望不一致。

| task | question 摘要 | prediction 表头 | gold 表头 | 值 |
| --- | --- | --- | --- | --- |
| task_24 | Women's Soccer attendance count | `attendance_count` | `COUNT(T2.link_to_member)` | `17` 正确 |
| task_26 | Physics Teaching members count | `COUNT(*)` | `COUNT(T2.member_id)` | `1` 正确 |
| task_27 | member total cost | `SUM(cost)` | `SUM(T2.cost)` | `866.25` 正确 |
| task_74 | eye colour | `eye_colour` | `colour` | `Brown` 正确 |
| task_86 | race names | `race` | `name` | 16 行均正确 |
| task_249 | average up votes and age | `AVG(UpVotes)`, `AVG(Age)` | `AVG(T1.UpVotes)`, `AVG(T1.Age)` | 均正确 |
| task_350 | T-shirt medium count | `COUNT(*)` | `COUNT(T1.event_id)` | `7` 正确 |

根因：

- prompt 只要求 answer 工具接收 `columns` 和 `rows`，但没有强约束列名必须与底层 SQL/原始字段/评估口径一致。
- 模型倾向输出自然语言别名，如 `attendance_count`、`post_id`、`percentage_blue_eyed`。
- gold 文件保留了 SQL 表达式或原始字段名，当前 evaluator 对列名严格比较。

### 6.2 输出了多余列

这类任务常见于问题只要求一个字段，但 agent 把辅助字段也提交了。

| task | 问题 | prediction | gold | 影响 |
| --- | --- | --- | --- | --- |
| task_38 | list withdrawals in cash transactions | `trans_id,date,operation,amount` | `trans_id` | 140 行 ID 正确，但多了 3 列 |
| task_259 | highest score comment | `Id,PostId,Score,Text` | `Text` | Text 正确，但多了辅助列 |
| task_415 | constructor reference and website | `constructor_reference_name` | `constructorRef,url` | 只输出了 `mclaren`，漏掉 URL |
| task_355 | full name and cost | `member_full_name,cost` | `first_name,last_name,cost` | 姓名被合并成一列 |

根因：

- 模型没有在最终 answer 前做“只保留题目要求字段”的投影。
- 对“full name”等自然语言表达，模型会合并字段；gold 则可能要求拆分成原始列。
- 复杂任务中模型把调试/验证用字段一起提交，导致 schema 错误。

## 8. 修改建议

### 8.1 立即调整运行配置

1. 将 `configs/alibaba.yaml` 的 `run.max_workers` 从 `8` 降到 `1` 或 `2`，优先避免 429 和额度瞬时耗尽。
2. benchmark 前增加一次 API preflight，确认 key 未欠费、额度可用，再启动 50 任务。
3. 对失败任务支持 resume/retry，不要整批重跑。优先重跑 `task_352, task_396, task_408, task_418, task_420`。

### 8.2 增强模型 API 错误处理

建议修改 `src/data_agent_baseline/agents/model.py`：

1. 对 429/rate limit 做指数退避重试，并记录 request id。
2. 对 400 `Arrearage`、`insufficient_quota` 这类不可通过短时间重试解决的问题，快速失败并给出明确的 billing/quota 分类。
3. 增加显式 request timeout 配置，避免单个 API 请求长时间占住任务。

### 8.3 强化最终答案 schema 约束

建议修改 `src/data_agent_baseline/agents/prompt.py` 的系统提示：

1. 最终 `answer` 只能包含题目要求的列，不允许提交调试字段、join key、日期、operation、score 等辅助列。
2. 如果题目要求多个字段，必须按 gold/原始字段粒度拆列；例如 first name 和 last name 不要合并成 full name。
3. 聚合列尽量使用底层表达式或 SQL 输出列名，避免自然语言别名导致严格 evaluator 判错。
4. 对 count/average/percentage/sum 不要四舍五入，除非题目明确要求格式化。

### 8.4 增加 answer 前本地校验

建议在 agent 提交前增加一个轻量 validation/reflection 步骤：

1. 检查列数是否等于题目要求的输出字段数。
2. 检查是否输出了明显的中间字段，例如 `CustomerID`、`date`、`operation`、`Score`。
3. 对 “lowest/highest/min/max” 检查并列值，避免只输出第一行。
4. 对 “percentage/average” 保留完整浮点字符串。
5. 对 “list countries/names/ids” 做 `DISTINCT` 和最终投影确认。

### 8.5 使用 SQL-first 模板处理结构化任务

大量错误来自 join、聚合、筛选、最终投影。建议为常见模式增加固定 playbook：

1. `count`：输出一行一列，列名跟 SQL 表达式保持一致。
2. `average/percentage/ratio`：保留完整 Python/SQL 浮点结果，不手工格式化。
3. `min/max`：先算极值，再回查所有并列行。
4. `list`：最终只投影题目要求列，必要时 `DISTINCT`。
5. `group/tally`：输出聚合结果，不输出明细表。

### 8.6 改进评估报告口径

当前 evaluator 只给 strict exact match，容易把“值正确但列名错”和“值错”混在一起。建议新增辅助指标：

1. `value_only_match`：忽略表头，仅比较 rows。
2. `projection_contains_gold`：prediction 多列时，判断 gold 行是否可由 prediction 投影得到。
3. `numeric_tolerant_match`：对浮点数使用容差比较。
4. `schema_error_type`：区分列名错误、多余列、缺列、行数错误。

这样可以更快定位是 agent 推理问题，还是输出格式问题。

## 9. 优先级建议

推荐按以下顺序修：

1. **先处理 API 额度/并发**：把 `max_workers` 降到 1-2，增加 429/backoff 和欠费快速失败；否则 hard/extreme 后半段会持续缺失 prediction。
2. **再处理输出 schema**：这是最大收益项，15 个任务只是列名问题，若修好可从 22% 提升到至少 52% 的值级正确。
3. **再处理最终投影和行数校验**：可覆盖 task_38、task_257、task_259、task_355、task_415 等格式类错误。
4. **最后处理复杂语义推理**：包括 task_169、task_200、task_344 等真正计算错误，需要更强 SQL 计划和中间结果验证。

## 10. 结论

本次 run 的严格正确率为 **22%**。主要失败并非单一原因：

- 后半段 5 个任务因 DashScope 额度/欠费错误没有运行完成。
- 15 个任务计算值正确但输出列名不符合严格 evaluator。
- 多个任务输出了中间表或多余列，没有做最终投影。
- 少数任务存在真实筛选、聚合、精度或 join 逻辑错误。

最有性价比的改动是：降低并发并增强 API 重试、强化最终 answer schema、禁止四舍五入、为 min/max/list/group/count/percentage 增加 SQL-first 校验模板。
