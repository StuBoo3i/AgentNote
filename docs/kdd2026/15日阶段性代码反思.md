## 1. 总体执行顺序

当前项目的单任务执行主链路如下：

```text
run-task / run-benchmark
  -> runner._run_single_task_core()
    -> 读取 PublicTask
    -> build_model_adapter()
    -> create_default_tool_registry()
    -> LangGraphReActAgent.run()
      -> LangGraph graph:
         START
           -> profile_context
           -> build_plan
           -> plan_action
           -> execute_action
           -> plan_action / validate_answer / END
           -> validate_answer
           -> END
      -> AgentRunResult
    -> 写 trace.json / prediction.csv / failure payload
```

四个部分在链路中的先后关系：

```text
1. LangGraph 基准模型框架
   负责整体状态机、prompt 构造、工具调用、answer 提交和 trace 产出。

2. Task Context Pack
   在 profile_context 阶段构建，进入 bootstrap observations、planning prompt、ReAct prompt 和 answer validation。

3. unifiedDB
   在 profile_context 阶段由 LangGraph 调用 build_unified_db(force=True) 预构建。
   后续 agent 可通过 inspect_unified_schema / execute_unified_sql 查询。

4. DocSage-inspired doc structuring
   被 Task Context Pack 用来生成 doc_schema_hypotheses / doc_extraction_requirements。
   被 unifiedDB 用来把 doc-extracted tables 导入统一 SQLite。
   也被工具 registry 暴露为 inspect_doc_schema / build_doc_tables / execute_doc_sql。
```

这四部分不是并列独立流程，而是嵌套增强关系：

```text
LangGraph 是执行骨架。
Task Context Pack 是确定性上下文压缩和答案契约。
unifiedDB 是结构化数据统一查询层。
DocSage-inspired 模块为 doc/md 提供候选结构化表，并进一步进入 unifiedDB 或独立 doc SQL 工具。
```

## 2. 第一部分：基于 LangGraph 的基准模型

### 2.1 基础原理

当前 LangGraph 部分的核心目标是把原始 ReAct 循环拆成显式状态机：

- 先 profile context。
- 再生成 high-level plan。
- 然后循环执行 `plan_action -> execute_action`。
- 如果模型提交 `answer`，进入 `validate_answer`。
- 如果超过 `max_steps` 或出现不可恢复失败，结束。

代码位置：

- `src/data_agent_baseline/agents/langgraph_agent.py`
- `src/data_agent_baseline/run/runner.py`

配置入口：

- `src/data_agent_baseline/config.py`
- `runner._run_single_task_core()`

`runner._run_single_task_core()` 根据 `config.agent.framework` 创建 agent：

```text
framework == "langgraph" -> LangGraphReActAgent
framework == "react"     -> ReActAgent
```

LangGraph 配置来自：

- `config.agent.max_steps`
- `config.agent.enable_langsmith`
- `config.langgraph.context_max_depth`
- `config.langgraph.context_inspection_file_limit`
- `config.langgraph.context_inspection_sample_rows`
- `config.langgraph.context_inspection_max_chars`
- `config.langgraph.planning_context_char_budget`
- `config.langgraph.execution_context_char_budget`
- `config.langgraph.enable_answer_validation`
- `config.langgraph.require_supported_answer`
- `config.langgraph.enable_context_pack`
- `config.langgraph.context_pack_char_budget`
- `config.langgraph.prompt_total_char_budget`
- `config.langgraph.prompt_recent_step_count`
- `config.langgraph.prompt_max_bootstrap_observations`

### 2.2 运行逻辑与流程

`LangGraphReActAgent._build_graph()` 定义节点：

```text
START
  -> profile_context
  -> build_plan
  -> plan_action
  -> execute_action
  -> continue: plan_action
  -> validate: validate_answer
  -> end: END
validate_answer
  -> END
```

关键 state 字段由 `AgentGraphState` 定义，包括：

- `task`
- `steps`
- `step_index`
- `max_steps`
- `answer`
- `failure_reason`
- `context_profile`
- `context_summary`
- `task_context_pack`
- `task_snapshot`
- `skill_definitions`
- `bootstrap_observations`
- `high_level_plan`
- `answer_validation`
- `pending_raw_response`
- `pending_prompt_messages`

#### profile_context

代码：`LangGraphReActAgent._node_profile_context()`

执行内容：

1. 调用 `list_context` 工具读取 context 文件树。
2. 根据 entries 构建 `context_profile`。
3. 对部分文件做轻量 inspection，生成 `file_summaries`。
4. 调用 `_build_unified_db_profile(task)`，即 `build_unified_db(task, force=True)`。
5. 构建 `context_summary`。
6. 如果 `enable_context_pack=True`，调用 `build_task_context_pack()`。
7. 将以下内容写入 bootstrap observations：
   - `context_schema_summary`
   - `inspect_unified_schema`
   - `task_context_pack`
   - `doc_schema_plan`
   - `skill_recommender`

这里是四部分首次汇合的位置：

```text
LangGraph profile_context
  -> unifiedDB profile
  -> Task Context Pack
  -> Doc schema plan summary
```

#### build_plan

代码：`LangGraphReActAgent._node_build_plan()`

该节点调用模型生成 high-level plan。prompt 由 `_build_plan_messages()` 构造。

planning prompt 明确要求：

- 使用 Task Context Pack 作为 primary planning context。
- 区分 `source_map.output_field_sources` 与 `source_map.filter_field_sources`。
- filter-only source 不能投影为最终答案列。
- output 和 filter 分属不同 source 时使用 `join_keys`。
- 参考 `answer_contract` 的 expected value slots、row grain、aggregation grain、ranking null_policy、joins、match policies。
- 如果 `doc_extraction_requirements` 非空，plan 必须说明 doc table、join key、filter field 和 answer grain。

#### plan_action

代码：`LangGraphReActAgent._node_plan_action()`

该节点：

- 如果 `step_index >= max_steps`，直接设置失败。
- 否则构造 ReAct prompt，调用 `model.complete(messages)`。
- 不直接执行工具，只保存 `pending_raw_response`。

#### execute_action

代码：`LangGraphReActAgent._node_execute_action()`

该节点：

1. 调用 `parse_model_step(raw_response)` 解析模型输出。
2. 如果 action 是 `answer`：
   - 先调用 `guard_answer_action_input()` 做基础答案格式修正和约束。
   - 再调用 `_context_pack_source_errors()` 检查 source contract。
   - 如果 source contract 有错误，作为 recoverable error 写入 step，不提交 answer。
3. 如果 action 是普通工具：
   - 通过 `ToolRegistry.execute()` 执行。
   - 记录 `StepRecord`。
4. 根据 answer / failure / max_steps 路由。

#### validate_answer

代码：`LangGraphReActAgent._node_validate_answer()`

该节点调用 `_validate_and_normalize_answer()`，同时接入 Task Context Pack 相关 warning / error：

- 列数和 answer_contract.expected_columns 是否一致。
- 是否疑似漏掉 expected answer slot。
- 是否投影 filter-only 字段。
- 是否使用了 projection source conflict 中的错误同名字段。
- 是否违反 metric source / filter scope / precision policy。
- 最终 SQL 是否可见 required join。

### 2.3 在项目执行链路中的位置

LangGraph 是主控层，负责：

- 建立任务上下文。
- 调用 deterministic profiler。
- 调用模型生成计划和动作。
- 调用工具。
- 产出 trace。
- 控制 max_steps。
- 提交 prediction。

其他三个增强模块都不能独立完成任务，必须通过 LangGraph 的 prompt、工具调用和 validation 被模型使用。

### 2.4 代码实现深度

当前 LangGraph 实现已经不只是简单 ReAct 包装，至少包含以下工程化能力：

- 显式 StateGraph。
- profile / planning / execution / validation 分阶段。
- bootstrap observations。
- high-level planning prompt。
- prompt 预算配置。
- repeated tool error notice。
- near max_steps answer 提醒。
- answer guard。
- context-pack-aware answer validation。
- LangSmith metadata 支持。
- trace metadata 中保留 `task_context_pack`、`high_level_plan`、`answer_validation`。

仍需注意的边界：

- repeated tool error notice 当前主要针对失败 step 的 error signature；对“工具成功但结果为空、模型反复查询空表”的硬阻断有限。
- near max_steps 当前是 prompt 级提醒，不是强制提交已有候选。
- high-level plan 是模型生成，不是 deterministic planner；计划是否真正锁定答案契约仍依赖 prompt 和后续 validation。

## 3. 第二部分：Task Context Pack

### 3.1 基础原理

Task Context Pack 是当前项目中面向 context 文件阅读和任务理解的确定性压缩层。

代码位置：

- `src/data_agent_baseline/agents/context_pack.py`

入口函数：

```text
build_task_context_pack(task, context_profile, file_summaries, context_root)
```

核心思想不是再次调用模型，而是用确定性规则从问题、结构化文件 profile、knowledge.md、unifiedDB schema、doc schema plan 中抽取：

- 题意类型。
- 输出字段。
- 过滤字段。
- join 关系。
- answer contract。
- doc extraction requirements。
- validation checks。

### 3.2 运行逻辑与流程

`build_task_context_pack()` 的实际执行顺序：

```text
1. profile_structured_sources()
2. 将 context_profile["unified_db"] 注入 data_profile
3. infer_question_intent()
4. extract_relevant_knowledge_facts()
5. link_question_to_schema()
6. infer_join_keys()
7. infer_answer_contract()
8. plan_doc_schema()
9. _doc_extraction_requirements()
10. _unresolved_schema_questions()
11. build_execution_plan()
12. build_validation_checks()
13. 返回 pack
```

输出 pack 的主要字段：

```text
question_intent
source_map
answer_contract
doc_schema_hypotheses
unresolved_schema_questions
doc_extraction_requirements
knowledge_facts
data_profile
execution_plan
validation_checks
pack_metadata
```

### 3.3 关键数据结构

#### question_intent

由 `infer_question_intent()` 生成。

代码中可见它会识别：

- operation type。
- expected answer kind。
- mentioned entities。
- field mentions。
- quoted values。
- sort / tie rules。
- derived metrics。

实际字段以函数返回为准；后续又会补充：

```text
question_intent["output_fields"]
question_intent["filters"]
```

#### source_map

由 `link_question_to_schema()` 生成。

代码中后续会使用：

- `output_field_sources`
- `filter_field_sources`
- `filter_only_sources`
- `join_keys`

该结构直接影响 planning prompt 和 answer validation。

#### answer_contract

由 `infer_answer_contract()` 生成，是当前 Task Context Pack 最重要的约束对象。

代码中可见字段包括：

```text
expected_kind
expected_columns
numerator
denominator
ratio
filter_scope_contracts
metric_source_contracts
precision_policy
filters_must_apply
forbidden_projection_fields
forbidden_projection_sources
projection_source_conflicts
row_grain
metric_grain
filters
sort
joins
match_policies
warnings
scoring_note
```

特别重要的实现点：

- `expected_columns` 使用 qualified source，例如 `json_drivers.number`。
- `projection_source_conflicts` 用于处理多个表中同名字段冲突。
- `metric_source_contracts` 用于防止把用户级 metric 替换成 post/detail 级 metric。
- `filter_scope_contracts` 用于防止过滤条件被错误扩大。
- `precision_policy` 用于防止未要求时 rounding。
- `filters_must_apply` 用于保证题目中必须应用的 filter 不被遗漏。
- `scoring_note` 明确记录：header 和列顺序不关键，完整 unordered data value vectors 关键。

#### doc_schema_hypotheses / doc_extraction_requirements

`build_task_context_pack()` 会调用 `plan_doc_schema()`。

之后 `_doc_extraction_requirements()` 根据题目关键词和 doc schema columns 判断是否必须结构化 doc。

当前代码中明确覆盖的 doc 任务类型包括：

- commander / legal / content warning -> legalities。
- advertisement / budget / meeting -> budget。
- superhero / height / publisher / marvel -> superhero。
- patient / creatinine / fibrinogen / white blood / wbc / age -> patient/laboratory。
- grand prix / race / constructor / champion -> race。

注意：这些是代码里看到的启发式类型，不应扩写成代码未实现的通用 IE 能力。

### 3.4 在项目执行链路中的位置

Task Context Pack 在 `profile_context` 阶段构建，并进入三个位置：

1. `bootstrap_observations`
   - trace 中能看到 `task_context_pack` 和 `doc_schema_plan`。

2. planning prompt
   - `_build_plan_messages()` 中作为 primary planning context。

3. ReAct execution prompt
   - `_build_messages()` 中作为 primary source-mapping and validation context。

4. answer validation
   - `_context_pack_source_errors()`
   - `_context_pack_answer_warnings()`

因此 Task Context Pack 的作用不是直接执行查询，而是约束模型计划、工具选择和最终答案。

### 3.5 代码实现深度

当前 Task Context Pack 已经实现了较深的“题意契约”能力：

- 文件 profile。
- schema linking。
- knowledge fact extraction。
- join key inference。
- answer contract。
- 同名字段冲突识别。
- doc extraction requirement。
- unresolved threshold 标记。
- validation checks 生成。

但仍有边界：

- 大量规则是启发式字符串匹配，非完整语义解析。
- `infer_answer_contract()` 中存在不少领域/案例驱动规则，后续应持续检查是否过强。
- doc extraction requirement 判断依赖关键词和 columns 交集，可能漏掉表达方式变化。
- 医学 normal/abnormal 阈值当前倾向标记 unresolved，不做高置信硬编码。

## 4. 第三部分：unifiedDB

### 4.1 基础原理

unifiedDB 的目标是把一个 task 的结构化 context 文件统一导入一个 SQLite 数据库，降低模型跨 CSV / JSON / DB 查询的难度。

代码位置：

- `src/data_agent_baseline/tools/unified_db.py`

核心入口：

```text
build_unified_db(task, force=False)
inspect_unified_db(task)
inspect_unified_schema(task)
execute_unified_sql(task, sql, limit=200)
```

缓存位置：

```text
/tmp/dabench_unified/<digest>/<task_id>/unified.db
```

当前缓存版本由 `_UNIFIED_CACHE_VERSION` 参与 digest。

### 4.2 运行逻辑与流程

`build_unified_db()` 的执行顺序：

```text
1. 创建 task cache dir
2. 如 force=True 且已有 db，删除旧 db
3. 如果 db 已存在，直接 inspect_unified_db()
4. 遍历 task.context_dir 下所有文件
5. 导入 .csv
6. 导入 .json
7. 导入 .db / .sqlite
8. 调用 _copy_doc_extracted_tables() 导入 doc-extracted tables
9. 写入 _source_files
10. 写入 _field_catalog
11. 构造 _join_candidates
12. 返回 UnifiedDbArtifacts
```

### 4.3 CSV 导入

函数：

```text
_import_csv()
```

实现要点：

- 表名：`csv_<file_stem>`，经 `_unique_name()` 去重和规范化。
- 使用 `csv.DictReader`。
- 先读取最多 200 行用于类型推断。
- 再重新读取全量数据，按 batch 1000 写入 SQLite。
- 字段写入 `_field_catalog`，包含：
  - `table_name`
  - `column_name`
  - `original_column_name`
  - `source_path`
  - `inferred_type`
  - `sample_values`

### 4.4 JSON 导入

函数：

```text
_json_table_payloads()
_import_json()
```

当前支持的 JSON 形态：

- dict 且含 `records: list[dict]`。
- list[dict]。
- dict 中多个 key 对应 list[dict]。

导入规则：

- 表名：`json_<raw_table_name>`。
- dict/list cell 会转成 JSON string。
- 推断类型使用最多 200 行 sample。
- 写入 `_field_catalog`。

注意：当前 `_json_table_payloads()` 直接 `json.loads(path.read_text(...))`，这意味着 unifiedDB 的 JSON 导入不是 streaming；如果后续出现超大 JSON，这里仍是潜在风险点。

### 4.5 SQLite DB 导入

函数：

```text
_import_sqlite_db()
```

实现要点：

- 以只读 URI 打开原 DB。
- 枚举非 `sqlite_%` 表。
- 表名：`db_<db_stem>_<source_table>`。
- 读取 sample 200 行推断类型。
- 再全量复制，每 1000 行 batch insert。
- 写入 `_field_catalog`。

### 4.6 doc-extracted tables 导入

函数：

```text
_copy_doc_extracted_tables()
```

该函数调用：

```text
build_doc_tables(task, force=force)
```

然后把 doc structuring 生成的 SQLite 表复制到 unified DB。

当前重要行为：

- 如果 doc structuring 失败，在 `_source_files` 记录 `source_type = doc_extracted_error`。
- 如果 doc table reported row_count <= 0，不导入实体表，只记录 `source_type = doc_extracted_empty`。
- 如果实际 rows 为空，也 drop 掉已创建表，并记录 `doc_extracted_empty`。
- 非空 doc table 作为 `source_type = doc_extracted` 导入。
- doc table 字段同样进入 `_field_catalog`。

这意味着 unifiedDB 当前不只是 CSV/JSON/DB，也会 best-effort 包含 doc-extracted tables。

需要注意：`registry.py` 中 `execute_unified_sql` 的工具描述仍写着 text documents are not imported into this DB；但当前 `unified_db.py` 实际会导入 doc-extracted tables。这个描述和代码事实存在轻微不一致，后续可修正。

### 4.7 schema inspection 与 SQL 执行

`inspect_unified_schema()` 返回：

- `path`
- `tables`
- `source_files`
- `join_candidates`

如果 table 来自 `doc_extracted`，会加：

```text
extraction_note = "candidate table extracted from doc/*.md with evidence/confidence columns"
```

`execute_unified_sql()`：

- 只允许 `SELECT` / `WITH`。
- 禁止 attach / insert / update / delete / drop / create / alter / replace / vacuum / pragma。
- 默认 limit 200。
- 返回 columns、rows、row_count、truncated。

### 4.8 在项目执行链路中的位置

unifiedDB 在两个位置被使用：

1. 自动 profile：

```text
LangGraph profile_context
  -> _build_unified_db_profile()
    -> build_unified_db(task, force=True)
```

2. 工具调用：

```text
inspect_unified_schema
execute_unified_sql
```

agent 计划和执行 prompt 都鼓励：

- CSV/JSON/DB 场景优先使用 unified DB。
- joins / filters / aggregation / ordering / final projection 优先走 SQL。

### 4.9 代码实现深度

当前 unifiedDB 已经具备：

- CSV 全量导入。
- JSON 多形态导入。
- SQLite DB 全量复制。
- 类型推断。
- source file 元数据。
- field catalog。
- join candidate 推断。
- doc-extracted tables 导入。
- read-only SQL 防护。
- task-level 缓存。

边界：

- JSON 读取不是 streaming。
- join candidates 是启发式，不等于真实外键。
- doc-extracted tables 是候选证据表，不等于 gold truth。
- 空 doc table 已降权，但低质量非空 doc table 仍可能误导模型，需要 confidence/coverage 进一步使用。

## 5. 第四部分：DocSage-inspired 非结构化文档结构化

### 5.1 基础原理

该部分借鉴的是“先把非结构化 doc/md 中与任务有关的实体、属性、关系抽成候选 relational tables，再用 SQL 推理”的思路。

代码位置：

- `src/data_agent_baseline/tools/doc_structuring.py`
- `src/data_agent_baseline/tools/registry.py`

当前实现是 deterministic / rule-based，不是完整 DocSage 论文系统，也没有看到额外 LLM-based schema discovery。

核心入口：

```text
plan_doc_schema(task, context_pack=None, unified_schema=None)
extract_doc_records(task, doc_schema_plan=None)
build_doc_tables(task, doc_schema_plan=None, force=False)
inspect_doc_tables(task)
execute_doc_sql(task, sql, limit=200)
```

缓存位置：

```text
/tmp/dabench_unified/<digest>/<task_id>/doc_structured.db
```

当前缓存版本由 `_DOC_STRUCT_CACHE_VERSION` 参与 digest。

### 5.2 schema planning 流程

函数：

```text
plan_doc_schema()
```

执行顺序：

```text
1. 遍历 task.context_dir 下 .md / .txt 文件
2. 跳过 knowledge.md
3. 每个文件读取前 12000 chars sample
4. 调用 _doc_table_kind(question, relative_path, sample)
5. 根据 kind 生成 doc table schema
6. 生成可能 join_keys
7. 如果传入 unified_schema，调用 _align_doc_join_keys() 对齐真实 unified DB 字段
8. 返回 tables / join_keys / uncertainties / metadata
```

当前 `_doc_table_kind()` 能识别的 kind：

- `legalities`
- `budget`
- `superhero`
- `race`
- `patient`
- `laboratory`
- `record`
- `entity`
- fallback generic facts

### 5.3 extraction 流程

函数：

```text
extract_doc_records()
```

执行顺序：

```text
1. 如果没有 doc_schema_plan，先 plan_doc_schema(task)
2. 逐个 table 读取 source doc
3. _split_chunks() 按 markdown heading / chunk 分段
4. 根据 entity 选择 extractor
5. 计算 coverage
6. 返回 extracted tables
```

extractor 映射：

```text
legalities -> _extract_legalities()
budget     -> _extract_budget()
superhero  -> _extract_superhero()
race       -> _extract_races()
patient    -> _extract_patients()
laboratory -> _extract_laboratory()
record     -> _extract_rec_facts()
entity     -> _extract_entity_facts()
fallback   -> _extract_generic_facts()
```

所有结构化 row 会尽量携带 provenance：

```text
_source_path
_chunk_id
_evidence
_confidence
```

### 5.4 doc SQLite 写入与查询

函数：

```text
build_doc_tables()
_write_doc_tables()
inspect_doc_tables()
execute_doc_sql()
```

写入内容：

- 每个 doc table 一张实体表。
- `_doc_source_files` 记录 source_path、table_name、entity、row_count、coverage_json。
- `_doc_field_catalog` 记录 table_name、column_name、inferred_type、role、source_path。

`execute_doc_sql()` 和 unified SQL 一样只允许 read-only SELECT / WITH。

### 5.5 工具注册

`registry.py` 暴露以下工具：

- `inspect_doc_schema`
- `build_doc_tables`
- `execute_doc_sql`
- `inspect_structured_context`

`inspect_structured_context` 会同时返回：

- unified schema。
- doc schema plan。
- doc tables。

这些工具允许模型在 ReAct 阶段主动把 doc 转成候选表并查询。

### 5.6 在项目执行链路中的位置

DocSage-inspired 模块有三条进入执行链路的路径：

#### 路径 A：Task Context Pack

```text
build_task_context_pack()
  -> plan_doc_schema()
  -> doc_schema_hypotheses
  -> doc_extraction_requirements
```

该路径主要用于 planning 和 validation。

#### 路径 B：unifiedDB 自动导入

```text
build_unified_db()
  -> _copy_doc_extracted_tables()
    -> build_doc_tables()
      -> extract_doc_records()
  -> doc tables copied into unified.db
```

该路径让 doc-extracted tables 出现在 `inspect_unified_schema()` 和 `execute_unified_sql()` 中。

#### 路径 C：agent 工具显式调用

```text
inspect_doc_schema
build_doc_tables
execute_doc_sql
inspect_structured_context
```

该路径允许模型在发现 doc source 必需时主动构建和查询 doc 表。

### 5.7 代码实现深度

当前 DocSage-inspired 模块已经实现：

- markdown/txt 文件遍历。
- heading/chunk 分段。
- query-related schema plan。
- 多种实体类型抽取。
- join key 初步推断和 unified schema 对齐。
- evidence/confidence。
- coverage。
- SQLite 写入。
- doc SQL 查询。
- unifiedDB 集成。
- 工具注册。

边界：

- 不是 LLM extraction。
- 不是完整 Interactive Schema Discovery。
- 抽取 schema 仍由文件名、题目关键词、正则和 sample 文本决定。
- 当前 extractor 覆盖的是项目失败案例中出现的高频形态，不是任意非结构化文档通用解析。
- 低置信非空 doc table 仍可能被模型误用；后续应加强 coverage/confidence 的使用方式。

## 6. 四部分之间的交互细节

### 6.1 profile_context 是关键汇合点

`profile_context` 同时完成：

```text
list_context
file inspection
unifiedDB build
Task Context Pack build
doc schema plan bootstrap
skill recommendation
```

因此后续优化如果影响上下文读取、统一数据库、doc structuring 或 context pack，优先检查 `LangGraphReActAgent._node_profile_context()` 的输入输出。

### 6.2 Task Context Pack 依赖 unifiedDB profile

`build_task_context_pack()` 接收的 `context_profile` 中已经包含：

```text
context_profile["unified_db"]
```

然后写入：

```text
data_profile["unified_db"]
```

这意味着：

- unifiedDB 的表名、列名、join_candidates 会影响 context pack 的 source mapping。
- 如果 unifiedDB 引入错误 doc table，context pack 可能跟着产生错误 source contract。
- 如果 unifiedDB 缺失某个 doc-extracted table，context pack 仍可能通过 `plan_doc_schema()` 看到 doc_schema_hypotheses，但 SQL 执行路径可能拿不到对应表。

### 6.3 DocSage-inspired 模块同时服务 Task Context Pack 和 unifiedDB

`plan_doc_schema()` 被 Task Context Pack 使用。

`build_doc_tables()` 被 unifiedDB 使用。

这意味着 doc schema plan 和 doc table extraction 必须保持一致：

```text
schema plan columns
  -> extractor row keys
  -> SQLite physical columns
  -> unifiedDB field catalog
  -> context_pack answer_contract / validation
```

如果其中任一环节字段名不一致，就会出现：

- plan 说有某字段，但 SQL 表中没有。
- answer contract 要求某 source，但 query 不可执行。
- validation 错误拦截本来正确的答案。

### 6.4 registry 是模型可见能力边界

模型实际能调用什么，由 `create_default_tool_registry()` 决定。

当前与四部分有关的工具：

```text
list_context
read_csv
read_json
read_doc
inspect_sqlite_schema
execute_context_sql
inspect_unified_schema
execute_unified_sql
inspect_doc_schema
build_doc_tables
execute_doc_sql
inspect_structured_context
execute_python
profile_schema
ground_fields
controlled_query 系列
answer
```

因此即使某个模块内部实现了函数，如果没有注册为工具，模型不能在 ReAct 循环中直接调用。

### 6.5 react.py 是模型输出格式兼容层

`parse_model_step()` 和 `_normalize_action_input()` 处理模型响应。

当前已看到的兼容能力：

- SQL 工具支持 `action_input` 是字符串。
- SQL 工具如果顶层 payload 有 `sql`，会补进 `action_input.sql`。
- `execute_python` 如果顶层 payload 有 `code`，会补进 `action_input.code`。
- `execute_python` 要求非空 code，否则抛出可恢复错误。
- `guard_answer_action_input()` 会检查答案 columns / rows、单值问题列数、最近工具表头对齐、聚合 alias 等。

这层对防止工具格式错误导致 max_steps 空转很关键。

## 7. 当前实现的阶段性优点

从代码阅读看，当前项目已经从“纯 LLM 看文件 + Python”逐步升级为多层 deterministic scaffold：

1. LangGraph 把执行拆成可追踪阶段。
2. Task Context Pack 把题意、source、join、answer contract 显式化。
3. unifiedDB 把结构化文件合并成统一 SQL 查询空间。
4. DocSage-inspired 模块把部分 doc/md 文件转成候选表。
5. answer validation 开始检查 source、join、列数、粒度、precision。

这套结构的核心价值是：

```text
降低模型自由发挥空间，
减少反复读长文件，
减少手写 Python/regex，
把跨文件 join/filter/aggregation 转成 SQL，
把最终答案从“看起来合理”变成“契约可检查”。
```

## 8. 当前实现的主要风险和待优化点

以下风险均来自代码阅读，不代表所有任务都会触发。

### 8.1 prompt 约束多，硬执行约束少

很多行为目前是 prompt instruction：

- 近 max_steps 提交已有候选。
- doc table 为空时 fallback read_doc。
- plan 必须包含 doc source。
- low-confidence mapping 需要验证。

但代码级强制较少。后续如果要提高稳定性，可以优先把高置信规则从 prompt 迁移到 deterministic repair / blocker。

### 8.2 Task Context Pack 规则有案例驱动倾向

`infer_answer_contract()` 中存在不少明确语义规则，例如：

- driver number。
- event type。
- lowest cost。
- average reading score。
- Formula1 race number。
- per unit。
- not 70 yet。
- post/user last editor。

这些规则提升了已知失败案例，但需要持续检查是否对其他任务形成过强先验。

优化原则应是：

```text
只在字段 source 高置信、问题语义明确、join 可验证时硬约束；
否则写入 warning / unresolved，而不是强制改写答案。
```

### 8.3 unifiedDB 与 registry 工具描述存在不一致

`unified_db.py` 当前会导入 doc-extracted tables。

但 `registry.py` 中 `execute_unified_sql` 描述仍写：

```text
Text documents such as doc/*.md and knowledge.md are not imported into this DB.
```

严格说，原文 doc 没有直接导入，但 doc-extracted candidate tables 会导入 unified DB。该描述容易让模型误判 doc table 是否能在 unified DB 中查询。

后续应改成：

```text
Raw doc text is not imported; successful doc-extracted candidate tables may appear with evidence/confidence columns.
```

### 8.4 JSON unified 导入不是 streaming

`_json_table_payloads()` 当前直接 `json.loads(path.read_text(...))`。

如果 hidden task 有很大 JSON，仍可能带来内存或耗时风险。此前 CSV 导入已是 streaming/batch，但 JSON 仍不是。

### 8.5 doc structuring 仍是有限规则系统

当前 doc structuring 的 schema types 是有限集合。

如果文档出现新领域或不同表达：

- `_doc_table_kind()` 可能落入 generic facts。
- extractor 可能生成低覆盖表。
- unifiedDB 可能导入非空但低价值表。

后续应重点增强：

- confidence / coverage 对 prompt 和 validation 的影响。
- 空表/低覆盖表的自动降权。
- schema plan 与 extractor 的一致性检查。

### 8.6 source validation 依赖 latest SQL 可见性

`_context_pack_source_errors()` 主要从最近 SQL action 检查 source、join、rounding 等。

如果模型通过 Python 计算或多步 SQL 中间表完成，部分 source contract 可能无法被准确验证。

后续可以考虑：

- 对 execute_python 的 code 做更结构化的 source usage extraction。
- 在工具 observation 中记录 result provenance。
- 对 final answer 绑定最近候选表 provenance。

## 9. 后续 Codex 优化时的建议阅读顺序

为了减少理解偏差，建议后续修改前按以下顺序读代码：

```text
1. runner.py
   确认 agent 创建、config 传递、timeout、trace/prediction 写出。

2. langgraph_agent.py
   确认 StateGraph 节点、prompt、validation、source blocker。

3. context_pack.py
   确认 answer_contract、source_map、doc requirements 是否过强或缺失。

4. unified_db.py
   确认表名、列名、source_files、field_catalog、join_candidates、doc table 导入。

5. doc_structuring.py
   确认 doc schema plan、extractor、coverage、SQLite 写入。

6. registry.py
   确认模型实际可调用工具和工具描述是否与实现一致。

7. react.py
   确认模型输出格式兼容、answer guard 是否阻断正确答案。
```

## 10. 后续优化的最低风险原则

基于当前代码结构，后续优化应遵守：

1. 先确认失败是否是当前代码 bug：
   - schema 错。
   - extractor 漏。
   - source contract 误拦。
   - cache 没失效。
   - prompt 描述与代码不一致。

2. 如果是缺失处理逻辑，再增量优化：
   - 优先增强确定性 profile / schema / validation。
   - 避免 task-id 特判。
   - 避免无证据领域常量。
   - 保持失败时可 fallback，而不是硬阻断。

3. 所有新增规则都应落到可验证对象：
   - source path。
   - table.column。
   - join key。
   - evidence。
   - confidence。
   - answer_contract 字段。

4. 每次修改后至少验证：
   - `python -m compileall`
   - 相关函数级 smoke test。
   - 至少一个失败任务 trace 对应路径。
   - 变更记录追加到 `file-change-notes`。

## 11. 阶段性结论

当前项目的四部分已经形成可运行的分层架构：

```text
LangGraph 主控执行
  + Task Context Pack 题意和答案契约
  + unifiedDB 结构化数据统一查询
  + DocSage-inspired doc 候选表抽取
```

最关键的工程事实是：

- LangGraph 不是单纯循环，而是 profile -> plan -> execute -> validate 的状态机。
- Task Context Pack 已经进入 planning、execution 和 validation 三个环节。
- unifiedDB 已经预构建，并被作为优先 SQL 查询空间。
- doc structuring 不只是独立工具，也已通过 unifiedDB 融入统一查询空间。

下一阶段优化不应简单继续堆规则，而应优先检查：

- 四层之间的信息是否一致。
- 工具描述是否匹配真实实现。
- answer_contract 是否过强或过弱。
- doc table 的 confidence / coverage 是否真正影响使用策略。
- max_steps 前是否有代码级候选答案修复机制。

这些方向都能在当前代码中找到明确落点，适合后续 Codex 继续迭代。

   - 当前 final answer 检查不仅看 Task Context Pack，也会检查 context contract 中的 expected columns、required joins、ranking null policy、ratio / percentage 计算形态。

3. 重复空结果 / 成功但无效查询没有硬熔断。
